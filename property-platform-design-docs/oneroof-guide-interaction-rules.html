OneRoof Guide — Interaction Rules & Metrics
Agentic Chat — how the Guide behaves, and how we'll know it works
Design Challenge
Mari Kondrateva · April 2026
Recommended option: Agentic Chat. A conversational Guide drives the journey with streamed working steps, data-source pills (MoE, Maps, OneRoof), inline property cards and multi-turn follow-ups that re-rank in place. A Smart Search variant (NL populates filter; ranked list) was also prototyped as a lower-risk comparison — the rules below are written for Agentic Chat; filter search stays available as the always-on escape hatch.

Core Interaction Principles

✓
Suggest, don't decide.Every Guide output is a recommendation the user can accept, ignore, or override. Filter search is always one tap away.
✕
Never auto-act on behalf of the user.The Guide never sends enquiries, books viewings, or dismisses properties without explicit confirmation.
✓
Show reasoning, always.Every match has a "Why this matches" panel. Every insight cites its data source (OneRoof Estimate, council records, suburb profile, etc.).
✕
Never invent facts.If a data point is unavailable or uncertain, say so. No confident guesses about schools, flood zones, or commute times.
✓
Ask before learning.When the Guide detects a behavioural pattern, it asks for permission before applying it: "Should I prioritise single-level homes?"
✕
Never nudge on emotionally sensitive signals.Price drops on a user's saved home-they-missed, or listings in a rejected suburb — stay quiet.
✓
Escalate on uncertainty.If the Guide can't confidently answer ("I'm not sure — the school zone data for this address looks outdated"), it flags the gap rather than bluffing.
✕
Never replace agent contact.The Guide prepares users for agent conversations, never substitutes for them. "Enquire" always routes to the listing agent.

When the Guide Speaks vs. Stays Quiet

Situation Guide Behaviour Rationale
User types a natural-language query Open chat; stream working steps inline with data-source pills (MoE, Maps, OneRoof); deliver property cards in the thread Conversational surface — transparency is the UI
User asks a follow-up ("walk time to primary?") Re-rank the existing shortlist in place using conversation state; don't re-search from scratch Multi-turn refinement is the Guide's signature capability
Guide is mid-stream and user taps Stop Halt the specialist immediately; leave completed working steps visible; surface partial results Interruption is a first-class interaction, not an error
User opens a property detail page Render Guide Insights card with contextual data (school zone, commute, price, suburb signals) Reduces cognitive load at the decision moment
User switches to filter search Silent. Small "Ask Guide" entry point stays in the search bar Filter search is a first-class path; don't force AI
User dismisses 3+ suggestions in a row Pause proactive nudges for 7 days; keep reactive features active Low acceptance = signal to back off
Low-confidence match or sparse data Show confidence signal ("Likely match — limited data") or omit the recommendation Better no recommendation than a bad one

Transparency & Trust Requirements

What the user always sees

  • What the Guide is doing right now (streamed status messages from Router)
  • Which data source backs each insight (OneRoof Estimate, CoreLogic, council, suburb profile)
  • A reasoning panel on every ranked match ("Why this matches")
  • Confidence level when it's not high ("Limited data for this property")
  • A clear path to edit, refine, or reject the Guide's interpretation

What the user can always do

  • Switch to filter search at any point, losing nothing
  • Ask "why?" on any recommendation and get a plain-English answer
  • Give feedback (thumbs up/down) that immediately shapes future output
  • Turn off proactive nudges while keeping reactive Guide features
  • See what the Guide has learned about them (preferences page)
  • Delete learned preferences at any time
On hallucinations and property data. Property decisions involve hundreds of thousands of dollars. An agent that invents a school zone or misreports a flood risk creates real harm. Every Guide output must be traceable to a concrete data source in OneRoof's systems (listings, CoreLogic, suburb profiles, council records, sold data). If a specialist agent can't back a statement, it doesn't make the statement. This is a hard product rule, not a nice-to-have.

Failure Modes & Recovery

Failure User-facing behaviour Recovery
Router can't classify intent Ask clarifying question with 2–3 suggested interpretations as chips User tap refines; log for training
Specialist times out Fall back to filter search with pre-filled criteria from the query User gets something useful; no dead end
Zero results for natural language query Suggest loosening 1–2 constraints (e.g. "+10% on budget would show 8 more") Preserves user's original criteria
Insight contradicts visible listing data Defer to the listing source; flag internally for data-quality review Listings are source of truth over generated insights

OneRoof Guide — Success Metrics
What we measure and why it matters
Design Challenge
Mari Kondrateva · April 2026
Measurement philosophy. Vanity metrics (AI query volume, time in app) can be gamed and don't reflect value. The Guide is successful if it (1) helps buyers make decisions faster and with more confidence, (2) delivers more qualified leads to agents, and (3) earns user trust over time. Every metric below maps to one of those three outcomes.

Engagement Metrics — Buyer Value

Metric Definition Target Why it matters
Time-to-shortlist Median days from first search to 5+ saved properties −30% in 6mo Direct signal the Guide reduces the research burden
Weekly active "For You" users % of active searchers engaging with proactive nudges weekly 40% Indicates the Guide earns return visits, not just first use
Search-to-enquiry rate % of NL searches producing ≥1 agent enquiry, vs. filter-search baseline +20% vs baseline Better matching = more motivated enquiries
Return sessions per saved property Avg. sessions revisiting a saved property over 30 days 2× uplift Proactive nudges bring users back at the right moments
Shortlist-to-viewing rate % of saved properties where user attends open home or books viewing +15% Better shortlists = more real-world action

Lead Quality Metrics — Agent Value

Metric Definition Target Why it matters
Enquiry-to-viewing conversion Agent-reported % of Guide-sourced enquiries that attend a viewing +25% vs baseline Proves Guide leads are warmer, not just more numerous
Agent NPS on lead quality Quarterly survey of paying agents on enquiry quality +15 points Directly tied to willingness to pay for listings
Impression-to-enquiry ratio Listing views ÷ enquiries for Guide users vs. filter users 1.5× baseline Guide users browse with more intent

Trust & Product Health Metrics

Metric Definition Target Why it matters
Suggestion acceptance rate % of Guide proposals (e.g. "prioritise single-level?") accepted 40–70% Too low = noise. Too high = not being bold enough
Reversion rate % of users who switch back to filter search after using the Guide <15% Direct signal of whether the Guide earns its place
Explainability click-through % of users expanding "Why this matches" reasoning Monitor Healthy engagement with reasoning = trust being built
Negative feedback rate Thumbs-down rate on Guide outputs (searches, insights, nudges) <5% Primary model-tuning signal
Nudge opt-out rate % of users disabling proactive notifications within 30 days <10% High opt-out = nudges feel noisy or irrelevant
Data-source citation coverage % of Guide outputs with a traceable source in OneRoof systems 100% Non-negotiable — no hallucinated facts

Business & Cost Metrics

Revenue signals

  • Incremental paid-listing revenue — uplift in agency subscriptions tied to Guide-driven lead quality
  • Agent-profile premium conversions — agencies upgrading based on reported enquiry improvements
  • Home-loan partner conversions — users accepting Guide-initiated mortgage consultations

Cost & efficiency signals

  • Cost per qualified lead — inference spend ÷ agent enquiries generated. Trend down over time
  • Router-to-specialist handoff ratio — % of queries resolved by router alone vs. escalated. Inform caching strategy
  • Cache hit rate — % of suburb/school/commute queries served from cache
  • Latency at p50 / p95 — user-perceived response time. Target p95 under 3s

Reporting Cadence

  • Daily — latency, error rates, negative feedback, cost per query
  • Weekly — engagement metrics, suggestion acceptance, nudge opt-outs
  • Monthly — time-to-shortlist, reversion rate, cost trends
  • Quarterly — agent NPS, revenue impact, cohort analysis
Decision rules. Reversion rate >20% → pause rollout, investigate. Negative feedback on a specialist >10% → pull the agent, re-prompt, re-evaluate. Agent NPS drops after launch → commercial team reviews lead quality in person with top agencies. Guide features ship when they move metrics, not when they demo well.

OneRoof Guide — Open Questions
For the OneRoof squad to work through together
Design Challenge
Mari Kondrateva · April 2026
Why this page exists. The concept and the metrics are my best thinking at four hours in. The questions below are the ones I'd want to sit down and resolve with Product, Engineering, Data and Commercial before committing to a Phase 1 scope. Treating these as open — rather than guessing at them alone — is how I'd expect to operate as Design Lead.

Build Effort & Team Shape

Question Why it matters My current lean
What does the minimum viable Phase 1 look like? "Agentic Chat + Router + Search Specialist" could be 6 weeks or 6 months depending on how much of the listings pipeline we reuse vs. rebuild. Ship Ask mode only: one specialist agent, chat surface, data-source pills, multi-turn follow-up. Defer Learn (For You nudges, learned preferences) and full Guide (Compare) to P2/P3.
Who owns the agents — Product Engineering or a new AI/Platform function? Affects how prompts are versioned, how evals run in CI, who's on-call when a specialist misbehaves. OneRoof may not have an AI platform team yet. Start inside Product Engineering with a named tech lead. Spin out a platform function only when a second product (e.g. agent-facing tools) needs the same infrastructure.
What's the right team shape for Phase 1? Too small and we under-build the eval harness; too large and we over-engineer before we've learned anything. 1 PM, 1 Design Lead (me), 2 mobile engineers, 1 backend/AI engineer, 0.5 data analyst, 0.5 user researcher. Tight squad, 3-month horizon.
Model choice — Anthropic, OpenAI, Google, or multi-provider from day one? Switching costs are real. Router+specialist architecture makes swapping easier, but auth, observability and cost reporting all need to be provider-aware. Start with one provider end-to-end to ship fast. Abstract the model call at the agent boundary so swapping later is a config change, not a rewrite.
What's the content/data licensing position? Guide summaries draw from CoreLogic estimates, council records, suburb profiles, Herald Homes editorial. Licences may not cover AI-generated derivative use. Legal review before P1. Default to citing and linking out rather than summarising, until licensing is confirmed.

What Does Month 1 Actually Look Like?

Weeks 1–2 — Align & scope

  • Workshop with Product, Engineering, Data: agree Phase 1 scope, success metrics, kill criteria
  • Architecture spike with Engineering on Router + Specialist pattern; pick the first model provider
  • User research sprint: 6 interviews with first-home buyers on how they articulate search today
  • Design audit: inventory what reuses from the existing system, what's net-new (the "one atom / two molecules" list)

Weeks 3–4 — Prototype & eval harness

  • Functional prototype of Ask mode against the real listings API — staging environment
  • First 50 queries in the offline eval set, scored by a property expert panel
  • Moderated test with 6 users on the prototype — does the reasoning land? Is the transparency enough?
  • Design system PR: new tokens, match-% badge, reasoning tags, confidence chip, Router status line

What I'd expect to have by end of month 1

  • A scoped P1 — not "agentic search" but a named, metric-backed slice of it
  • A working prototype against real data, reviewable by stakeholders via a shared URL
  • An eval harness in CI so model/prompt changes are tested, not hoped
  • User evidence — 6 interviews + 6 prototype tests is small but enough to kill bad assumptions early
  • A design system delta that Engineering can review before build, not after
  • Commercial buy-in from the agent-facing side of the business — what do they need to see to trust this?

Risks I'd be watching from day one

  • Data quality. Hallucination risk scales with sparse or stale source data. Guide is only as good as the listings feed.
  • Agent trust. If agents perceive Guide as disintermediating them, they push back commercially. Early agency engagement is non-negotiable.
  • Cost creep. Router-first helps, but a lazy prompt in a specialist can 10× inference spend overnight. Daily cost monitoring from week 1.

Things I'd Want to Know from the OneRoof Team

  • Gut read on the chat surface? Does OneRoof's user base — first-home buyers especially — want to converse, or would they rather see a ranked list? (I'm backing chat on the 79% NZ conversational-AI adoption stat; interested where the panel lands.)
  • Where does the current app have its worst drop-off — search, detail, or enquiry? That's where the Guide earns its place first.
  • What's the real relationship with CoreLogic on data licensing and ML use of Estimates?
  • How much of the listings pipeline is real-time vs. batch? Affects how fresh the Guide's context can be.
  • What's the agent feedback loop today? Do we survey them, or only hear when they churn?
  • Is there existing telemetry I can look at for "For You" usage patterns before I design its evolution?
  • What's NZME's appetite for shared AI infrastructure across Herald, Driven, OneRoof — or is each product on its own stack?
  • What's the mobile app's test infrastructure? Can we run the 10% A/B on-device, or do we need server-side gating?
  • Where does Claude Design (or similar) sit with NZME's design/engineering tooling budget for the next 12 months?
A note on this page. A senior design lead's job isn't to arrive with all the answers — it's to arrive with the right questions and a view on how to resolve them. The items above are the ones I'd want on the whiteboard in week one, and they're the reason I'd push back if someone asked me to commit to a 12-month roadmap on day one.
